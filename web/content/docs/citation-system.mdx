---
title: Citation System
description: "How citations link wiki content back to contract documents"
---

On Wikipedia, citations exist for public verifiability. Here, the sources are contract documents. Citations serve a different purpose: traceability. When a future agent (or the project team) reads a claim and wants to find the original contract basis, the citation should point them to a specific, locatable document in the vault.

## MediaWiki's native ref system

All inline citations use `<ref>` tags, rendered via `<references />` at the bottom of the page in a `== References ==` section. This is standard MediaWiki and requires no custom extensions.

```wikitext
The clarifier walls use #5 rebar at 8" O.C. for penetrations greater than
6" diameter.<ref name="rfi-042">
{{Cite rfi|number=042|date=2026-01-22|hash=a1b2c3d4e5f6
|note=Wall reinforcement at penetrations}}</ref>
```

## Citation templates

Five templates for inline citations, matching common source types in construction projects. A sixth template, `{{Cite vault}}`, is used in the Bibliography section to describe full document sets consulted.

### Cite drawing

For facts derived from contract drawings.

```wikitext
{{Cite drawing|sheet=S-302|detail=7|date=2025-08-15
|hash=a1b2c3d4e5f6|note=Wall reinforcement detail at penetrations}}
```

Renders as: Drawing S-302, Detail 7 (hash a1b2c3d4)

### Cite spec

For specification requirements.

```wikitext
{{Cite spec|section=03 30 00|paragraph=2.1.A|date=2025-06-01
|hash=a1b2c3d4e5f6|note=Concrete mix requirements}}
```

Renders as: Spec 03 30 00, §2.1.A (hash a1b2c3d4)

### Cite rfi

For RFI questions and responses.

```wikitext
{{Cite rfi|number=042|date=2026-01-22|hash=a1b2c3d4e5f6
|note=Wall reinforcement at penetrations}}
```

Renders as: RFI-042, 22 January 2026

### Cite submittal

For approved submittal data.

```wikitext
{{Cite submittal|number=SUB-033000-001|date=2026-02-10
|hash=a1b2c3d4e5f6|note=Concrete mix design approval}}
```

Renders as: Submittal SUB-033000-001, 10 February 2026

### Cite geotech

For geotechnical report data.

```wikitext
{{Cite geotech|report=Geotechnical Investigation|boring=B-7
|date=2025-03-15|hash=a1b2c3d4e5f6|note=Groundwater elevation at clarifier site}}
```

Renders as: Geotech Report, Boring B-7, 15 March 2025

### Cite vault

For the Bibliography section. Describes a full document set consulted during analysis.

```wikitext
{{Cite vault|type=drawings|snapshot=a1b2c3d4e5f6
|timestamp=2025-08-15|note=Volume 3 — Civil Drawings, 45 sheets}}
```

Renders as: Vault: Volume 3 — Civil Drawings, Aug 2025 (snapshot a1b2c3d4)

### Common fields

All templates include:

- **hash** — the content-addressed vault hash of the source document (first 8 chars is fine for display)
- **date** — the date of the source document
- **note** — brief human-readable description of what this citation supports

`{{Cite vault}}` additionally includes:

- **type** — the kind of document set (drawings, specs, reports, etc.)
- **timestamp** — the date or date range of the documents

## Granularity

Not every sentence needs a citation. The goal is traceability, not exhaustiveness.

### Always cite

- Design parameters (dimensions, elevations, material requirements). These come from specific drawing details or spec paragraphs and need clear pointers
- Verbatim contract language in `{{Verbatim}}` blocks. Cite the specific spec section and paragraph
- RFI responses that modify original contract requirements. Cite the RFI number and response date
- Approved submittal data that updates equipment specifications. Cite the submittal number
- Claims that were corrected or superseded on the talk page. Cite the resolution source

### Don't need individual citations

- General descriptions of a process area's function that are evident from multiple drawings
- Information already attributed inline ("Per Spec 03 30 00, §2.1.A...") where the reference itself serves as the locator
- Content on drawing pages where the entire page analyzes a single drawing identified in the title

## Named refs for reuse

Many facts from the same drawing or spec paragraph share a source. Use named refs:

```wikitext
The influent pipe is 48 inches in diameter.<ref name="c-301" />
It enters the distribution box at elevation 102.5 ft.<ref name="c-301">
{{Cite drawing|sheet=C-301|date=2025-08-15|hash=a1b2c3d4e5f6
|note=Civil site plan with pipe routing}}</ref>
The distribution box splits flow to four clarifiers.<ref name="c-301" />
```

## Page structure

Every area page and equipment page ends with two sections:

```wikitext
== References ==
<references />

== Bibliography ==
{{Cite vault|type=drawings|snapshot=a1b2c3d4e5f6
|timestamp=2025-08-15|note=Volume 3 — Civil Drawings, 45 sheets}}
{{Cite vault|type=specs|snapshot=b2c3d4e5f6a1
|timestamp=2025-06-01|note=Project Specifications, Divisions 01-46}}
```

**References** are inline citations. Each one traces a specific claim back to a specific document in the vault. They are auto-generated from `<ref>` tags.

**Bibliography** lists the full document sets consulted for the page. This tells a reader (or agent) what contract documents the page drew from overall, without tying it to individual claims.

## Migration

Existing pages should add citations incrementally. An agent doesn't need to cite every claim in one pass. A reasonable approach:

1. **First pass** — cite all design parameters in discipline sections and all `{{Verbatim}}` blocks
2. **Second pass** — cite RFI responses and submittal data that modified original requirements
3. **Third pass** — cite anything flagged as corrected or superseded on the talk page

This can be spread across multiple tasks.
