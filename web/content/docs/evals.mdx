Evals run end-to-end: an agent receives a task, source data, and a fresh wiki instance, then produces pages across multiple checkpoints. Output is graded automatically against rubrics and a human-written reference.

## Protocol

The incremental eval uses a **6-checkpoint editorial workflow** that mirrors how a human editor would build a wiki page from raw data:

1. **Survey** — snapshot a source directory and create an enriched `Source:` page. Catalog media, transcribe voice notes, assess data quality and coverage gaps. Plan editorial approach.
2. **Draft** — write the person or episode page using only the first source. Include infobox, lead paragraph, sections with prose and citations. Note episode candidates and gaps on the Talk page.
3. **New source** — snapshot a second source directory, create its source page, then **revise** the existing article by weaving new data into appropriate sections (not appending). Cross-reference facts across sources. Update the Talk page.
4. **Episodes** — create episode pages for rich narratives (first meetings, trips, conflicts, milestones). Link from the main page with one-sentence summaries.
5. **Owner input** — integrate first-person testimony from the wiki owner. Cite using `{{Cite testimony}}`. Note conflicts between testimony and digital evidence on the Talk page.
6. **Verify** — final editorial review. Audit tone, balance, and gaps. Produce a citation manifest pairing every factual claim with its source evidence.

A fresh MediaWiki instance is provisioned per run with templates and no content pages. The agent's conversation is resumed between checkpoints so it can build on prior work.

## Graders

Graders are organized into three tiers that combine into a weighted composite.

### Quality tier (50%)

**Reference coverage** — compares output against a human-written reference page. Measures section heading overlap, infobox field coverage, citation hash coverage, and category overlap. Each dimension is weighted and combined into a single score.

**Accuracy** — resolves the citation manifest produced at the verify checkpoint. For each factual claim, checks whether the cited source evidence actually supports it. Claims are classified as supported, unsupported, or fabricated (contradicted by evidence). Fabrications carry a 2x penalty. Owner testimony claims cited with `{{Cite testimony}}` are classified as "attributed" rather than unsupported.

### Content tier (30%)

**Completeness** — checks structural elements: lead paragraph, infobox, body sections, `References`/`Bibliography` sections, categories, prose word count, subsection count, inline citations, blockquotes, and media embeds. Depth thresholds scale with checkpoint — earlier checkpoints have lower requirements.

**Editorial** — LLM-graded evaluation of Wikipedia editorial conventions: neutral third-person tone, past tense for biographical events, section organization, prose quality, wikitext syntax, and appropriate use of wikilinks. Uses multi-pass grading with median scoring.

**Integration** — measures revision quality when a page is updated across checkpoints. Checks whether new facts are placed in appropriate existing sections (not dumped in an "Additional information" block), whether the page reads as one unified article after revision, and whether conflicting data is noted on the Talk page. Only scored when the page has been revised from a previous checkpoint.

**Source criticism** — evaluates whether source pages include critical assessment beyond raw statistics. Checks for platform limitation notes, gap identification (missing date ranges, encrypted content), data quality assessment, querying instructions, and content breakdowns.

### Mechanics tier (20%)

**Citations** — validates citation templates (`{{Cite message}}`, `{{Cite vault}}`, `{{Cite testimony}}`, etc.) have required fields. Each template type has specific requirements: `snapshot` and `date` for digital sources, `speaker` and `date` for testimony. Score = valid / total.

**Cross-referencing** — measures whether the page combines information from multiple source types. Counts facts that synthesize data across sources vs. possible cross-references. Enabled when two or more source types are present.

**Tool usage** — checks harness logs for expected CLI tool usage: `wai snapshot`, `wai read`, `wai write`/`create`/`edit`. Binary per tool.

### Composite formula

```
page_composite = 0.5 × quality + 0.3 × content + 0.2 × mechanics
```

The overall composite splits 20% source pages (averaged) and 80% content pages. Content pages are weighted: person 85% + talk 15% (or person 50% + episodes 40% + talk 10% when episode pages exist).

## Observations

- **Calibration significantly changes rankings.** Raw accuracy scores ranged from 0.462 to 0.933. After removing false penalties (misnumbered citation targets, Talk page notes treated as unsupported claims), calibrated accuracy ranges from 0.846 to 0.990. Claude Code Opus 4.6 moved from 2nd to 1st overall; Codex GPT-5.2 moved from 5th to 3rd.
- **Accuracy measures manifest quality, not factual accuracy.** The accuracy grader checks whether the citation manifest correctly maps claims to source evidence. An agent can write a factually accurate article but score low if its manifest has incorrect evidence lookups. Calibration corrects the most egregious cases, but the grader still conflates source-linking precision with truthfulness.
- **Editorial scoring has high variance.** The editorial grader uses LLM evaluation and scores range from 0.000 (Kimi K2.5) to 0.900 (GPT-5.2). Multi-pass median scoring helps but editorial quality remains the most subjective dimension.
- **Cross-referencing and integration are near-ceiling.** Most agents score 1.000 on cross-ref and integration once they process both source types. These graders primarily catch failure modes (ignoring a source entirely, appending instead of weaving) rather than distinguishing quality.
- **Source pages cluster tightly.** All agents score 0.65–0.80 on source documentation. The main differentiator is source criticism depth — whether the agent notes platform limitations and data quality issues.
- **Speed varies 4x across models.** Kimi K2.5 finishes in 39 minutes vs. 2h 29m for Opus 4.6 on OpenCode. Faster runs correlate with lower scores but the gap narrows at the top — Opus 4.5 on Claude Code matches Opus 4.6's composite in under an hour.

## Known issues

- **Accuracy conflates manifest quality with factual accuracy**: agents that write correct articles but produce imprecise citation manifests are penalized. Splitting into separate factual-accuracy and citation-integrity graders would improve signal.
- **Editorial grader subjectivity**: LLM-based editorial evaluation shows high variance across passes and may penalize valid stylistic choices.
- **Draft checkpoint produces zero pages for some harnesses**: OpenCode agents sometimes skip page creation at the draft checkpoint, resulting in 0.000 scores that don't reflect the final output quality.
- **Episode page grading limited**: episode pages are graded as part of the person page composite but lack dedicated episode-specific rubrics.
